## Phase Boundary

This phase delivers three capabilities: (1) an AI-driven chaos monkey exploration mode that autonomously navigates and tests an app, (2) CI integration with GitHub Actions workflow generation and flakiness detection, and (3) distribution packaging for npm publish, MCP server, and Claude Code plugin.

This phase does NOT add new scanner types, modify reporting formats, or change the core CLI command structure. It builds on top of the existing browser runner, scanner registry, and AI infrastructure from Phases 1-3.

<decisions>
## Implementation Decisions

### Chaos Monkey Exploration

- **D-01:** Navigation uses intelligent crawling ‚Äî leverages existing route discovery (`src/analyzers/route-discoverer.ts`) and element extractor (`src/analyzers/element-extractor.ts`) to systematically visit pages, click interactive elements, and fill forms. Not random clicking.
- **D-02:** Form fill uses built-in edge-case data patterns ‚Äî hardcoded sets including empty strings, XSS payloads, SQL injection strings, Unicode edge cases, max-length strings, negative numbers, special characters. No external data dependency.
- **D-03:** AI decision trace is structured JSON log per action ‚Äî each exploration action records what element was targeted, what action was taken (click/fill/navigate), the AI's reasoning for choosing it, and what was observed after. Aligns with EXPLR-03 requirement for transparent decision traces.
- **D-04:** Exploration reuses the existing `BrowserRunner` and `PageHookPipeline` infrastructure ‚Äî console error monitoring, network failure detection, and screenshot capture all work during exploration without reimplementation.

### Flakiness Detection & Quarantine

- **D-05:** Test stability tracked via local JSON results history ‚Äî run results appended to `.sniff/history.json` (or configured path). Flake rate computed per test across last N runs. Zero infrastructure cost.
- **D-06:** Quarantine threshold defaults to 3+ failures in last 5 runs ‚Äî tests exceeding this are flagged as flaky, run but don't block CI pipeline exit code. Threshold configurable via `sniff.config.ts`.
- **D-07:** Flaky test detection is opt-in for `sniff run` but default-on for `sniff ci` mode ‚Äî local dev runs skip history tracking unless `--track-flakes` is passed, CI mode always tracks.

### CI Integration

- **D-08:** Single opinionated GitHub Actions workflow generated by `sniff ci` ‚Äî outputs `.github/workflows/sniff.yml` with Playwright browser setup, headless run, report artifact upload, and proper exit codes. One file, no matrix complexity for v1.
- **D-09:** Cache strategy uses GitHub Actions cache for Playwright browsers + node_modules ‚Äî standard `actions/cache` with playwright install path. Minimizes CI run time.
- **D-10:** CI mode (`sniff run --ci` or detected via `CI` env var) automatically enables headless, JUnit XML output, and non-zero exit on failure ‚Äî no separate `sniff ci-run` command needed beyond workflow generation.

### MCP Server

- **D-11:** Three MCP tools exposed: `sniff_scan`, `sniff_run`, `sniff_report` ‚Äî maps 1:1 to CLI commands. Simple mental model for AI agents.
- **D-12:** stdio transport ‚Äî standard MCP stdio protocol, compatible with Claude Desktop, Cursor, Windsurf, and other MCP clients out of the box.
- **D-13:** MCP server is a separate entry point (`src/mcp/server.ts`) but shares all core logic ‚Äî no code duplication, just a different interface layer on top of the same scan/run/report pipeline.

### Distribution & Packaging

- **D-14:** npm publish as primary distribution ‚Äî package.json already has `bin`, `files`, `exports` configured. Needs `publishConfig`, proper `keywords`, `repository`, `license` fields, and README for npm listing.
- **D-15:** Claude Code plugin uses plugin manifest format ‚Äî `plugin.json` manifest with tool definitions that invoke the sniff CLI under the hood. Installation via `claude plugin add sniff`.
- **D-16:** `npx sniff` must work for zero-install usage ‚Äî the package must handle Playwright browser installation gracefully on first run (prompt or auto-install).

### Claude's Discretion

- Exploration step ordering and prioritization algorithm (breadth-first vs depth-first page traversal)
- Specific GitHub Actions runner OS and Node version matrix defaults
- MCP tool parameter naming and description text
- npm package keywords and README structure
- Whether to use `@modelcontextprotocol/sdk` or implement stdio protocol directly

- `BrowserRunner` + `PageHookPipeline`: Exploration can reuse the entire browser infrastructure ‚Äî launch browser, create context, hook into page events
- `ScannerRegistry`: If exploration produces findings, they can flow through the existing scanner/finding pipeline
