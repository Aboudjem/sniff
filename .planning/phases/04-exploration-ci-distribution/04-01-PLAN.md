# Plan 04-01: Flakiness Detection + History Persistence

**Phase:** 04 - Exploration + CI + Distribution
**Requirements:** CI-03, CI-04
**Depends on:** None (builds on existing persistence layer, no Phase 4 inter-dependencies)

## Goal

Implement flakiness detection that tracks test stability across runs in `.sniff/history.json`, computes flake rates per test, and quarantines flaky tests so they run but do not block CI exit codes. This is the foundation that Plan 04-03 (CI Integration) depends on for its exit-code logic.

## Context

- `src/core/persistence.ts` -- Has `saveResults`/`loadLastResults` using `.sniff/` directory pattern with `mkdir(dir, { recursive: true })`, `readFile`, `writeFile`. New functions follow the same pattern.
- `src/core/types.ts` -- Has `Finding`, `BrowserFinding`, `Severity` types. New `TestRunRecord` and `FlakinessHistory` types go here.
- `src/config/schema.ts` -- Zod schema composition with `z.object({...}).optional()` pattern for each config section. New `flakiness` section follows this.

### Task 1: Types, Flakiness Algorithm, and Persistence Functions
**Type:** implementation
**Files:**
- Modify: `src/core/types.ts` -- Add `TestRunRecord` and `FlakinessHistory` interfaces
- Create: `src/core/flakiness.ts` -- `computeFlakeStatus` algorithm and `buildTestId` helper
- Modify: `src/core/persistence.ts` -- Add `appendRunHistory` and `loadFlakinessHistory` functions

**Implementation:**

Add to `src/core/types.ts` (append after `BrowserFinding`):

```typescript
export interface TestRunRecord {
  runId: string;
  timestamp: string;
  testId: string;        // stable: "${scanner}::${ruleId}::${url}"
  passed: boolean;
  duration: number;
  viewport?: string;
}

export interface FlakinessHistory {
  version: 1;
  runs: TestRunRecord[];
  flaky: string[];       // testIds currently above quarantine threshold
}
```

Create `src/core/flakiness.ts`:

- Export `buildTestId(finding: Finding | BrowserFinding): string` -- returns `"${finding.ruleId}::${finding.filePath}"` for source findings, or `"${finding.ruleId}::${(finding as BrowserFinding).url}"` for browser findings (check for `url` property). Per D-05, the ID must be deterministic across runs. Do NOT use `finding.message` (too variable).
- Export `buildTestRunRecords(results: ScanResult[], runId: string): TestRunRecord[]` -- iterates all findings across all scan results, creates a `TestRunRecord` per finding with `passed: false` (findings = failures). For tests that passed (no finding), we do not have positive signal in current architecture -- document this limitation with a comment.
- Export `computeFlakeStatus(history: FlakinessHistory, testId: string, windowSize?: number, threshold?: number): { isFlaky: boolean; failureCount: number; runCount: number }` -- per D-06, defaults: `windowSize=5`, `threshold=3`. Filter `history.runs` to matching `testId`, take last `windowSize` entries. If fewer than `windowSize` records exist, return `isFlaky: false` (not enough data). Count failures (`!r.passed`). Return `isFlaky: failureCount >= threshold`.

Extend `src/core/persistence.ts`:

- Add `const HISTORY_FILE = 'history.json';` and `const MAX_HISTORY_PER_TEST = 10;`
- Import `computeFlakeStatus` from `./flakiness.js` and `TestRunRecord`, `FlakinessHistory` from `./types.js`
- Add `appendRunHistory(rootDir: string, records: TestRunRecord[]): Promise<void>` -- reads existing history from `.sniff/history.json` (fallback to empty `{ version: 1, runs: [], flaky: [] }`), appends new records, trims to `MAX_HISTORY_PER_TEST` per testId using a `Map<string, TestRunRecord[]>` grouping, recomputes `flaky` list by calling `computeFlakeStatus` for each testId, writes back.
- Add `loadFlakinessHistory(rootDir: string): Promise<FlakinessHistory | null>` -- reads and parses `.sniff/history.json`, returns `null` on any error (file not found, parse error).

   const { buildTestId } = await import('../../core/flakiness.js');

     failOnSeverities.includes(f.severity) && !flakyTestIds.has(buildTestId(f))

## Threat Model

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Filesystem -> history.json | Untrusted JSON read from disk; could be malformed or tampered |
| CLI flags -> run behavior | User-controlled `--track-flakes` and `--ci` flags change exit behavior |

### STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-04-01 | Tampering | history.json | mitigate | Validate JSON structure on load; if parse fails, treat as empty history (graceful degradation). Do not execute code from the file -- only deserialize as data. |
| T-04-02 | Denial of Service | appendRunHistory | mitigate | Enforce MAX_HISTORY_PER_TEST (10) trim per testId to prevent unbounded file growth. Cap total records at MAX_HISTORY_PER_TEST * unique testIds. |
| T-04-03 | Information Disclosure | history.json | accept | File contains test IDs and pass/fail status only -- no secrets, credentials, or PII. Stored locally in `.sniff/` which is typically gitignored. |
| T-04-04 | Elevation of Privilege | CI exit code bypass | mitigate | Quarantine only suppresses exit code for tests meeting the flakiness threshold. A newly-failing test always blocks until it has sufficient history. Threshold requires `windowSize` consecutive runs minimum. |
</threat_model>
