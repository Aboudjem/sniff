# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3: Browser Runner + Quality Scanners + Reporting. This phase has two UI surfaces: terminal output (CLI) and a self-contained HTML report. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none (pure HTML/CSS for report, picocolors for terminal) |
| Icon library | none (ASCII severity icons in terminal: X, !, ~, -, i) |
| Font | System font stack for HTML report: `-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, 'Helvetica Neue', Arial, sans-serif` |

**Rationale:** Sniff is a CLI tool distributed via npm. No React/Next.js frontend. The HTML report must be a single self-contained file with no external dependencies (D-12). Terminal output extends the existing picocolors-based formatter.

---

## UI Surfaces

This phase produces two distinct output surfaces:

### Surface 1: Terminal Output (CLI)

Extends the existing `src/cli/formatter.ts` pattern. Used by `sniff run` for live progress and final results.

### Surface 2: HTML Report

Self-contained single-file HTML report (D-12). All CSS inline in a `<style>` block. All screenshots embedded as base64 data URIs. No JavaScript required for core functionality (progressive enhancement only for collapsible sections).

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline code padding, icon gaps in HTML report |
| sm | 8px | Compact list item spacing, tag padding |
| md | 16px | Default element spacing, card padding |
| lg | 24px | Section padding within report cards |
| xl | 32px | Section gaps between scanner groups |
| 2xl | 48px | Major section breaks (e.g., between severity groups) |
| 3xl | 64px | Report header/footer padding |

Exceptions: none

**Terminal spacing:** Use single blank lines between severity groups, double blank lines before the summary. Indent findings with 2 spaces (matches existing formatter pattern).

## Typography

### HTML Report

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (regular) | 1.5 | Finding descriptions, fix suggestions |
| Label | 12px | 600 (semibold) | 1.4 | Severity badges, scanner type tags, metadata |
| Heading | 20px | 600 (semibold) | 1.2 | Section headings (severity groups, scanner sections) |
| Display | 28px | 600 (semibold) | 1.2 | Report title, overall score |

**Code snippets:** `'SF Mono', 'Fira Code', 'Cascadia Code', Consolas, monospace` at 12px (reuses Label size), weight 400, line-height 1.6. Code blocks share the 12px size for dense, compact rendering consistent with the four-size scale above.

### Terminal Output

Terminal output inherits the user's terminal font. No font specification needed. Formatting uses:
- Bold (`pc.bold`) for summary counts and section headers
- Dim (`pc.dim`) for file paths, timestamps, and metadata
- Normal weight for finding messages

## Color

### HTML Report Color Palette

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` (light) / `#0D1117` (dark) | Page background, report surface |
| Secondary (30%) | `#F6F8FA` (light) / `#161B22` (dark) | Finding cards, code blocks, table stripes |
| Accent (10%) | `#2F81F7` | Report header links, "View Details" expand toggles, scan timestamp |
| Destructive | `#DA3633` | Critical/high severity badges only |

**Additional semantic colors for severity (used in badges and left-border accents on finding cards):**

| Severity | Badge BG | Badge Text | Card Left Border |
|----------|----------|------------|------------------|
| Critical | `#DA3633` | `#FFFFFF` | `#DA3633` |
| High | `#DA3633` | `#FFFFFF` | `#DA3633` |
| Medium | `#D29922` | `#1C2128` | `#D29922` |
| Low | `#3FB950` | `#1C2128` | `#3FB950` |
| Info | `#8B949E` | `#1C2128` | `#8B949E` |

**Accent reserved for:** Report header links, expand/collapse toggles on finding detail sections, the "last scanned" timestamp, and in-report navigation anchors. Never used for severity indication.

**Color scheme detection:** HTML report uses `prefers-color-scheme` media query to switch between light and dark palettes. Default is light.

### Terminal Color Mapping

Extends the existing `severityColors` in `formatter.ts`:

| Severity | Color Function | Existing |
|----------|---------------|----------|
| Critical | `pc.red` | Yes |
| High | `pc.red` | Yes |
| Medium | `pc.yellow` | Yes |
| Low | `pc.cyan` | Yes |
| Info | `pc.gray` | Yes |

**New terminal colors for Phase 3:**

| Element | Color Function | Purpose |
|---------|---------------|---------|
| Scanner name header | `pc.blue` + `pc.bold` | Distinguish scanner sections (a11y, visual, perf, e2e) |

## Copywriting Contract

### Terminal Output Copy

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable (CLI output, no interactive CTA) |
| Command start | `Running tests across {N} pages in {viewport}...` |
| Scanner progress | `[{scanner}] Scanning {url}...` |
| Test pass | `All tests passed. No issues found.` |
| Test fail summary | `Found {N} issues across {M} pages: {severity breakdown}` |
| Empty state (no tests) | `No generated tests found. Run 'sniff scan' first to analyze your codebase and generate tests.` |
| Empty state (no pages) | `No routes discovered. Ensure your app is running at the configured baseUrl.` |
| Error: browser launch | `Failed to launch browser. Ensure Playwright browsers are installed: npx playwright install chromium` |
| Error: app not reachable | `Could not reach {baseUrl}. Is your development server running?` |
| Error: lighthouse fail | `Performance measurement failed for {url}. Skipping performance scan for this page.` |
| Viewport labels | `desktop (1280x720)` / `mobile (375x667)` / `tablet (768x1024)` |

### HTML Report Copy

| Element | Copy |
|---------|------|
| Report title | `Sniff QA Report` |
| Report subtitle | `Generated {timestamp} | {N} pages scanned | {M} issues found` |
| Empty state heading | `Clean Bill of Health` |
| Empty state body | `No issues found across {N} pages. Either your app is flawless or sniff missed something. Run again with --verbose to be sure.` |
| Section: E2E | `Browser Tests` |
| Section: A11Y | `Accessibility` |
| Section: Visual | `Visual Regression` |
| Section: Performance | `Performance` |
| Section: Source | `Source Code` |
| Severity group header | `{severity}: {count} issues` (e.g., "CRITICAL: 3 issues") |
| Fix suggestion prefix | `Fix:` followed by the specific suggestion |
| Screenshot caption | `Failure screenshot: {test name} on {viewport}` |
| Visual diff caption | `Diff: {page} on {viewport} ({N}px changed, {percent}% difference)` |
| Performance over budget | `LCP {actual}ms exceeds budget of {budget}ms by {delta}ms` |
| A11Y violation | `{rule}: {description} ({impact} impact). Affects {count} elements.` |
| Report footer | `Generated by sniff v{version}. No sugar-coating, no excuses.` |
| Brutal tone example | Use direct language: "This page is broken" not "This page may have issues". "Fix this before shipping" not "Consider addressing this". |

### Destructive Actions

| Action | Confirmation Approach |
|--------|----------------------|
| `sniff update-baselines` | CLI prompt: `This will overwrite {N} baseline images. Continue? [y/N]` (default No) |
| `sniff update-baselines --yes` | Skip confirmation (for CI usage) |

**Tone guidelines (RPT-06):** The report must be brutally honest. Use active voice, direct statements, and specific numbers. Never hedge with "might", "could", or "consider". Examples:
- YES: "3 images have no alt text. Screen readers cannot describe these to users."
- NO: "Some images might be missing alt text. Consider adding alt attributes."
- YES: "LCP is 4200ms. Your users are staring at a blank screen for 4 seconds."
- NO: "LCP could be improved. The current value exceeds recommended thresholds."

## HTML Report Layout

### Structure

```
+--------------------------------------------------+
| HEADER: Sniff QA Report                          |
| Subtitle: timestamp | pages | issues             |
| [Summary badges: Nx critical, Nx high, ...]      |
+--------------------------------------------------+
| NAV: Jump links by section                        |
| [Browser Tests] [Accessibility] [Visual] [Perf]  |
+--------------------------------------------------+
| SECTION: Browser Tests                           |
|   SEVERITY GROUP: Critical                        |
|     FINDING CARD (left-border colored)            |
|       [severity badge] [viewport tag] [url]       |
|       Description text                            |
|       Fix: suggestion                             |
|       [Screenshot thumbnail -> expandable]        |
|     FINDING CARD...                               |
|   SEVERITY GROUP: High                            |
|     ...                                           |
+--------------------------------------------------+
| SECTION: Accessibility                           |
|   (same card pattern)                             |
+--------------------------------------------------+
| SECTION: Visual Regression                       |
|   DIFF CARD                                       |
|     [Baseline] [Current] [Diff overlay]           |
|     Pixel count and percentage                    |
+--------------------------------------------------+
| SECTION: Performance                             |
|   METRICS TABLE per URL                           |
|     LCP | FCP | TTI | Budget | Status             |
+--------------------------------------------------+
| FOOTER: Generated by sniff v{version}            |
+--------------------------------------------------+
```

### Finding Card Anatomy

Each finding renders as a card with:
- **Left border:** 4px solid, colored by severity
- **Header row:** Severity badge (label style) + scanner type tag + viewport tag (if applicable) + URL
- **Body:** Finding message at body size
- **Fix row:** "Fix:" prefix in semibold + suggestion text
- **Screenshot row** (if applicable): Thumbnail at 320px width, click/tap expands to full size via `<details>` element
- **Code snippet** (if applicable): Monospace block at 12px with syntax highlighting via inline CSS classes

### Visual Diff Card Anatomy

For visual regression findings:
- **Three-column layout:** Baseline | Current | Diff (overlay with changed pixels highlighted in `#FF00FF`)
- **Below images:** "{N}px changed ({percent}% of viewport)"
- **Images:** Responsive, max-width 100%, displayed at native resolution up to 400px per column

### Performance Table Anatomy

Per-URL performance table:
- **Columns:** Metric | Value | Budget | Status
- **Status column:** Green checkmark for pass, red X for fail
- **Over-budget rows:** Entire row gets light red background (`#DA363310`)

### Responsive Behavior

The HTML report must be readable at 320px viewport width (developer might open it on a phone):
- Finding cards stack vertically
- Visual diff images stack vertically (baseline above current above diff)
- Performance table scrolls horizontally
- Navigation links wrap

No reusable UI components in the traditional sense. Instead, the phase produces these output modules:

## Interaction Contracts

### HTML Report Interactions

| Interaction | Mechanism | Behavior |
|-------------|-----------|----------|
| Expand screenshot | `<details>` + `<summary>` | Click thumbnail to reveal full-size screenshot. No JS required. |
| Jump to section | `<a href="#section-id">` | Smooth scroll to scanner section via CSS `scroll-behavior: smooth` |
| Expand finding details | `<details>` + `<summary>` | Code snippets and additional context hidden by default, expandable |
| Print | CSS `@media print` | Hide navigation, expand all `<details>`, remove background colors |

### Terminal Interactions

| Interaction | Mechanism | Behavior |
|-------------|-----------|----------|
| Progress update | Overwrite current line with `\r` | Show "Testing {url}... " then append "done ({N} findings)" |
| Baseline update confirmation | stdin prompt | `readline` interface, default to No, accept y/Y for yes |
| Interrupt | SIGINT handler | Graceful shutdown: close browser, save partial results, print summary of what completed |
